# HandyAndy — 설계 결정 기록

## 아키텍처

### Next.js + Python 에이전트 분리
**결정**: 웹앱(Next.js)과 파일 I/O(Python 에이전트)를 분리
**이유**: Next.js는 브라우저 환경이라 로컬 파일시스템 직접 접근 불가. Python 에이전트가 HTTP 서버로 로컬 파일 접근을 대리.

### 포트 자동 탐색 (3001~3005)
**결정**: 고정 포트 대신 3001~3005 범위에서 자동 탐색
**이유**: 포트 충돌 방지. 웹앱은 시작 시 ping으로 에이전트 포트를 찾는다.

### 멀티스레드 에이전트
**결정**: `ThreadingMixIn` 사용
**이유**: 단일 스레드일 때 API 호출(10~30초) 중 ping 요청이 블록되어 UI에서 "연결 끊김"으로 표시되는 문제 해결.

## 데이터 설계

### 프로젝트 경로 필드명: `root`
**결정**: `path` 대신 `root` 사용
**이유**: `path`는 의미가 모호하고 하위 경로와 혼동됨. `root`가 프로젝트 최상위 폴더임을 명확히 표현.

### 설정 중앙 관리 (handyandy_settings.json)
**결정**: 프로젝트 목록과 경로를 중앙 settings 파일에 저장
**이유**: 프로젝트 폴더별 설정 파일 방식은 앱이 어느 폴더가 프로젝트인지 알 방법이 없어 결국 중앙 목록이 필요. 복잡도를 줄이기 위해 처음부터 중앙 방식 채택.

### API 키 별도 파일 (handyandy_private.json)
**결정**: API 키는 settings와 분리
**이유**: gitignore로 API 키 유출 방지. settings.json은 공유 가능, private.json은 각 PC마다 별도 설정.

## 파일 명명

### 버전 체계: V001 + .0001 서브버전
**결정**: 기본 `V001.json`, 버전업 시 `V001.0001.json`, `V001.0002.json`
**이유**: V002처럼 주 버전을 올리면 같은 작업의 반복 시도인지 새 버전인지 구분 불가. 서브버전으로 동일 작업의 반복 생성을 구분.

### 프리셋 접두사: TPL_
**결정**: `TPL_*.json`
**이유**: 템플릿(Template)의 약자. `PRE_`(preset)보다 용도가 명확하고 짧음.

### 이미지 경로: prompts → images 치환
**결정**: JSON 경로에서 `/prompts/`를 `/images/`로, `.json`을 `.png`로 치환하여 이미지 경로 파생
**이유**: JSON과 이미지가 항상 같은 버전명을 가지도록 단일 소스(filePath)에서 파생. 버전업 시 불일치 방지.

## UI/UX

### 프롬프트 저장 / 이미지 생성 버튼 분리
**결정**: "저장 후 전송" 단일 버튼 → "프롬프트 저장" + "이미지 생성" 분리
**이유**: 이미지 생성 실패 시 JSON까지 재저장하는 비효율 제거. 이미지만 재생성 가능하도록.
**연결**: 이미지 생성은 마지막으로 저장된 경로(`lastSavedPath`)를 기준으로 동작. 저장 전 이미지 생성 버튼은 비활성화.

### 덮어쓰기 3단계 선택
**결정**: 같은 파일 존재 시 취소/버전업/덮어쓰기 3개 버튼 제공
**이유**: 실수로 덮어쓰는 사고 방지. 버전업으로 이전 데이터 보존 가능.

### 검색 범위
**결정**: presets/ 폴더만 제외하고 전체 프로젝트 루트 검색
**이유**: 초기에 `prompts/` 폴더만 검색했으나 `_assets/prompts/` 폴더가 포함 안 되는 문제 발생. presets 폴더는 설정 파일이므로 검색 제외.
